# Deferred work

An AI agent generates a response, sends it to the user, and returns. After that, you need to score the output, log token costs, notify your team, or wait days for user feedback. None of that should block the response or affect whether the run succeeds.

Deferred functions solve this. You call `defer()` inside your agent function, passing data to an independent function that runs after the parent finalizes. The parent returns immediately. The deferred run gets its own retries, concurrency, and step state.

---

## The mental model

Think of deferred work as "because of" vs "as part of." Steps are work that is part of a run. Deferred functions are work that happens because of a run.

A `step.run()` call is part of the parent. If it fails, the parent retries. If it's slow, the parent waits. The parent's success depends on every step completing.

A `defer()` call is not part of the parent. The parent doesn't wait, doesn't see a result, and doesn't care if the deferred function fails. The deferred run is a separate function with its own lifecycle.

This distinction matters for AI because model calls are expensive. If you add scoring as a step at the end of your function and the scoring step fails, the entire function retries, including the model calls you already paid for. With defer, the agent run succeeds based on the agent work alone.

---

## How deferred runs relate to the parent

Each `defer()` call creates a separate Inngest function run. The deferred run is linked to the parent in traces, so you can see the full picture: the parent run, its steps, and every deferred run it spawned.

Deferred runs are enqueued when the parent run finalizes (succeeds, fails, or is cancelled). They do not start during the parent's execution. This means the parent always completes first, and then the deferred work begins.

Multiple `defer()` calls in one parent all run in parallel after finalization. A failed deferred run does not affect the parent's status or trigger the parent's failure handler.

---

## Where defer fits in the AI toolkit

**Scoring.** Use defer to evaluate agent output after serving it. A deferred scorer can use LLM-as-judge (a cheaper model evaluating the primary model's output), run deterministic checks, or wait for user feedback via `step.waitForEvent()`. See [Build a deferred scorer](/docs/features/inngest-functions/steps-workflows/deferred-scoring).

**Cost tracking.** Defer a function that logs token usage, calculates cost per customer, and writes to your analytics platform. The agent doesn't wait for your analytics pipeline.

**Experiments.** Combine defer with `group.experiment()` to score different model variants independently. The experiment selects the variant, the agent serves the result, and a deferred scorer evaluates the output. See [Running experiments](/docs/features/inngest-functions/steps-workflows/running-experiments).

**Temporal signals.** A deferred function can use `step.waitForEvent()` to pause for days or weeks. Wait for a conversion event, a support ticket reopening, or a user clicking "helpful." Score the original run based on what happens downstream.

## Comparison with other approaches

| Approach | Blocks parent? | Independent retries? | Typed data? | Linked in traces? |
|----------|---------------|---------------------|------------|-------------------|
| More steps at the end | Yes | No | N/A | Yes |
| `onFailure` handler | No | Yes | No (only error + input) | Partial |
| Send events to other functions | No | Yes | No (manual serialization) | No |
| External eval platform | No | Depends | Depends | No |
| `defer()` | No | Yes | Yes (schema-validated) | Yes |
