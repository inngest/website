# Score a function run

Scoring lets you measure how well a function run performed. You attach a named score to a run or a specific step, and Inngest tracks it over time in traces and dashboards.

This is useful for anything where "did it work?" isn't a binary answer: AI accuracy, response relevance, user satisfaction, task completion rates.

<Callout variant="info">
  Scoring requires the v4 SDK. Install with `npm install inngest@latest`.
</Callout>

---

## Setup

`inngest.score()` works out of the box. To use the durable `step.score()` method, register `scoreMiddleware()` on your client:

```ts
import { Inngest } from "inngest";
import { scoreMiddleware } from "inngest/experimental";

export const inngest = new Inngest({
  id: "my-app",
  middleware: [scoreMiddleware()],
});
```

Without the middleware, `step.score()` is unavailable on the `step` object.

---

## Two ways to write a score

There are two methods, and the difference matters:

- **`step.score(id, options)`** writes a *durable* score. The write is wrapped in a step, so it runs exactly once and is not re-sent when the function replays or retries. It takes a step `id` (like `step.run()`) and can only be used inside a function with `scoreMiddleware()` enabled. Reach for this for standalone score writes in your function body.
- **`inngest.score(options)`** writes a *live* score. It is not memoized, so call it from inside a `step.run()` callback (where the enclosing step already handles idempotency) or from outside the function entirely by passing a `runId`. The target is inferred from the execution context unless you set `runId`/`stepId` explicitly.

---

## Score the current run

Inside a function, call `step.score()` with a step ID, a name, and a value.

```ts
export default inngest.createFunction(
  { id: "summarize-ticket", triggers: { event: "support/ticket.created" } },
  async ({ event, step }) => {
    const summary = await step.run("generate-summary", async () => {
      return await generateSummary(event.data.content);
    });

    const passed = await step.run("check-guardrails", async () => {
      return validateOutput(summary);
    });

    // Score this run based on whether guardrails passed
    await step.score("guardrail-pass", { name: "guardrail-pass", value: passed ? 1 : 0 });

    return { summary, passed };
  }
);
```

`step.score()` attaches the score to the current run. Use it when you know the outcome before the function finishes.

## Score a specific step

To attach a score to a step's trace rather than the run, call `inngest.score()` from *inside* that step's callback. The target step is inferred from the execution context, so you don't pass a `stepId`:

```ts
await step.run("call-model", async () => {
  const result = await callModel(prompt);
  const confidence = result.metadata.confidence;

  // Inferred from context: attaches to the "call-model" step
  await inngest.score({ name: "model-confidence", value: confidence });

  return result;
});
```

The score appears on the step in the trace view, not just the run.

If you want a *durable* step-scoped write — or you're writing it from the function body rather than inside the callback — use `step.score()` and name the step explicitly with `stepId`:

```ts
await step.score("score-confidence", {
  name: "model-confidence",
  value: confidence,
  stepId: "call-model",
});
```

## Score from outside a function

You can score any run from external code. Pass the `runId` and optionally a `stepId`.

```ts
await inngest.score({
  name: "user-feedback",
  value: 1,
  runId: "01ABC123...",
});

// Score a specific step in that run
await inngest.score({
  name: "accuracy",
  value: 0.9,
  runId: "01ABC123...",
  stepId: "generate-summary",
});
```

This is how you score based on signals that arrive after the function finishes: a user clicks "helpful," a support ticket gets resolved, a conversion happens downstream.

## Parameters

| Parameter | Where | Type | Required | Notes |
| --- | --- | --- | --- | --- |
| `id` | 1st arg of `step.score` | `string` | Yes (`step.score` only) | Step ID; unique within the run. Memoizes the score write. Not used by `inngest.score`. |
| `name` | options | `string` | Yes | Score label. Use consistent names across runs so they aggregate. |
| `value` | options | `number \| boolean` | Yes | See [Score values](#score-values) below. |
| `runId` | options | `string` | No | Required when scoring from outside a function; inferred from context otherwise. |
| `stepId` | options | `string` | No | Omit to attach the score to the run; set to attach it to that step's trace. |

See the [TypeScript reference](/docs/reference/typescript/v4/functions/scoring) for full type signatures.

## Score values

A score's `value` can be a `number` or a `boolean`:

- **Numbers** can use any range that fits your use case — `0`–`1` for percentages or pass rates, integers for counts (tokens, tool calls), or raw measurements (latency in ms). Use a consistent range per `name` so the dashboard aggregates cleanly.
- **Booleans** are convenient for pass/fail outcomes; `true`/`false` are recorded as `1`/`0`.

```ts
await step.score("guardrail-pass", { name: "guardrail-pass", value: true });
await step.score("latency", { name: "latency_ms", value: 920 });
```

---

## Where scores appear

Scores show up in two places:

**Trace views.** Each score is attached to the run or step it was scored against. You can see what a run scored alongside its execution timeline.

**Function dashboards.** Score aggregates over time show trends. A degrading score after a prompt change tells you something broke. A stable score after a model swap tells you the change is safe.
