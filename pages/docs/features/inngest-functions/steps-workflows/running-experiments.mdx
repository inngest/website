# Run experiments in AI pipelines

When you build AI features, you're constantly making bets: GPT-4o or Claude, a terse prompt or a detailed one, a single-shot call or a full RAG pipeline. Most teams test these by hand with `if/else` blocks, a random number generator, and some logging they'll later regret.

`group.experiment()` replaces that with a durable primitive. You declare named variants, pick one per run with a selection strategy, and every choice is recorded in your traces automatically. The selection is memoized like any other step, so retries and replays always run the same variant. No external feature-flag service, no A/B infrastructure, and no inconsistent results when a function replays.

The rest of this guide follows one experiment end to end: comparing two models for document summarization, watching it run, comparing the variants, and promoting the winner.

## Before you start

Experiments run inside Inngest functions, so you need an Inngest client and a serve handler. If you already have a working Inngest app, skip ahead. If not, the [quick start](/docs/getting-started/nextjs-quick-start) walks through the full setup; here's the minimum:

```ts
npm install inngest
```

```ts {{ filename: "inngest/client.ts" }}
import { Inngest } from "inngest";

export const inngest = new Inngest({ id: "my-app" });
```

Serve your functions over an HTTP endpoint (Next.js shown; other frameworks in the [serving guide](/docs/learn/serving-inngest-functions)):

```ts {{ filename: "app/api/inngest/route.ts" }}
import { serve } from "inngest/next";
import { inngest } from "../../../inngest/client";
import summarizeDocument from "../../../inngest/summarize-document";

export const { GET, POST, PUT } = serve({
  client: inngest,
  functions: [summarizeDocument],
});
```

Every example below imports that `inngest` client. The `experiment` helper is a separate named import from the `inngest` package.

## Split traffic between two models

Start with the bet you want to make. Here we send half the traffic to GPT-4o and half to Claude. Each model call is a variant, and `experiment.weighted()` controls the split.

```ts {{ filename: "inngest/summarize-document.ts" }}
import { experiment } from "inngest";
import { inngest } from "./client";

export default inngest.createFunction(
  {
    id: "summarize-document",
    triggers: { event: "document/uploaded" },
  },
  async ({ event, step, group }) => {
    const doc = await step.run("fetch-document", () =>
      fetchDocument(event.data.documentId)
    );

    const summary = await group.experiment("model-comparison", {
      variants: {
        gpt4o: () =>
          step.run("summarize-gpt4o", () =>
            callOpenAI({ model: "gpt-4o", prompt: `Summarize: ${doc.text}` })
          ),
        claude: () =>
          step.run("summarize-claude", () =>
            callAnthropic({ model: "claude-sonnet-4-20250514", prompt: `Summarize: ${doc.text}` })
          ),
      },
      select: experiment.weighted({ gpt4o: 50, claude: 50 }),
    });

    return summary;
  }
);
```

`group.experiment()` returns the result of whichever variant ran, so `summary` is just the summary. The selection happens inside a memoized step: if the function retries or replays, the same model is used every time. Weights are relative proportions, so `{ gpt4o: 50, claude: 50 }` and `{ gpt4o: 1, claude: 1 }` behave identically.

`group` is a handler argument Inngest injects alongside `event` and `step` — it's where `.experiment()` lives, and there's nothing to enable to use it. The helper functions in these examples (`fetchDocument`, `callOpenAI`, `callAnthropic`, and later `searchVectorStore`, `trackExperiment`, `launchDarkly`, and friends) are your own code, shown as placeholders for whatever your function actually does.

To capture AI metrics, wrap the model call with `step.ai.wrap()` instead of a plain `step.run()`. You get experiment metadata and AI metadata (tokens, latency) on the same trace:

```ts
const result = await group.experiment("model-comparison", {
  variants: {
    gpt4o: () =>
      step.ai.wrap("summarize-gpt4o", generateText, {
        model: openai("gpt-4o"),
        prompt: `Summarize: ${doc.text}`,
      }),
    claude: () =>
      step.ai.wrap("summarize-claude", generateText, {
        model: anthropic("claude-sonnet-4-20250514"),
        prompt: `Summarize: ${doc.text}`,
      }),
  },
  select: experiment.weighted({ gpt4o: 50, claude: 50 }),
});
```

## Promote the winner

Once the data favors one variant, you don't have to rip the experiment out of your code. Switch the strategy to `experiment.fixed()` to send all traffic to the winner while keeping the structure in place. That way you can re-open the experiment later, or add a new challenger, without rewriting the function.

```ts
const summary = await group.experiment("model-comparison", {
  variants: {
    gpt4o: () =>
      step.run("summarize-gpt4o", () =>
        callOpenAI({ model: "gpt-4o", prompt: `Summarize: ${doc.text}` })
      ),
    claude: () =>
      step.run("summarize-claude", () =>
        callAnthropic({ model: "claude-sonnet-4-20250514", prompt: `Summarize: ${doc.text}` })
      ),
  },
  select: experiment.fixed("claude"),
});
```

When you're ready to delete the losing variant entirely, you can. Until then, `fixed()` is the clean way to declare a winner.

## Keep users on a consistent variant

Splitting traffic randomly is fine for stateless work, but when users interact with an AI feature repeatedly you usually want them to get consistent behavior. `experiment.bucket()` hashes a value you provide — a user ID, an account ID, any stable key — so the same value always maps to the same variant, even across different runs.

```ts
const response = await group.experiment("assistant-model", {
  variants: {
    current: () =>
      step.run("current-model", () =>
        generateResponse({ model: "gpt-4o", messages: conversation })
      ),
    candidate: () =>
      step.run("candidate-model", () =>
        generateResponse({ model: "gpt-4o-mini", messages: conversation })
      ),
  },
  select: experiment.bucket(event.data.userId, {
    weights: { current: 90, candidate: 10 },
  }),
});
```

Here 10% of users get routed to the candidate model and stay there. This prevents the jarring experience of quality shifting between requests for the same person.

## Compare multi-step strategies

A variant isn't limited to a single call. Variant callbacks can contain multiple sequential steps, each individually retried and memoized. This is how you compare approaches that involve different amounts of work, like a single-shot prompt versus a retrieval-augmented pipeline.

```ts
const answer = await group.experiment("prompt-strategy", {
  variants: {
    single_shot: () =>
      step.run("single-shot", () =>
        callLLM({ prompt: `Answer this question: ${question}` })
      ),
    rag_pipeline: async () => {
      const chunks = await step.run("retrieve-context", () =>
        searchVectorStore(question, { topK: 5 })
      );
      const context = chunks.map((c) => c.text).join("\n\n");
      return await step.run("generate-with-context", () =>
        callLLM({
          prompt: `Using this context:\n${context}\n\nAnswer: ${question}`,
        })
      );
    },
  },
  select: experiment.weighted({ single_shot: 70, rag_pipeline: 30 }),
});
```

Only the selected variant's steps execute. The unselected variant never runs.

## Act on which variant was selected

By default you get the variant's result. Pass `withVariant: true` to get both the result and the name of the variant that ran, which you need for logging, analytics, or a downstream decision.

```ts
const outcome = await group.experiment("tone-test", {
  variants: {
    concise: () =>
      step.run("concise-prompt", () =>
        callLLM({ prompt: "Be brief. " + userQuery })
      ),
    detailed: () =>
      step.run("detailed-prompt", () =>
        callLLM({ prompt: "Be thorough and explain your reasoning. " + userQuery })
      ),
  },
  select: experiment.weighted({ concise: 50, detailed: 50 }),
  withVariant: true,
});

await step.run("log-experiment", () =>
  trackExperiment({
    experiment: "tone-test",
    variant: outcome.variant,
    responseLength: outcome.result.length,
  })
);
```

With `withVariant: true`, `group.experiment()` returns `{ result, variant }`: `result` is the return value of the variant that ran (typed as the union of your variants' return types), and `variant` is the selected variant's name as a string-literal union of your variant keys (here `"concise" | "detailed"`). This shape is the same for every selection strategy — `weighted`, `bucket`, `fixed`, and `custom`. Without `withVariant`, you get `result` on its own.

## Run multiple experiments in one pipeline

You can run independent experiments in a single function. Use `experiment.bucket()` with a composite key so each experiment assigns variants on its own.

```ts
export default inngest.createFunction(
  {
    id: "ai-document-pipeline",
    triggers: { event: "document/process" },
  },
  async ({ event, step, group }) => {
    const userId = event.data.userId;

    const extraction = await group.experiment("extraction-model", {
      variants: {
        structured: () =>
          step.run("structured-extract", () =>
            extractWithSchema(event.data.documentUrl)
          ),
        freeform: () =>
          step.run("freeform-extract", () =>
            extractFreeform(event.data.documentUrl)
          ),
      },
      select: experiment.bucket(`${userId}:extraction`),
      withVariant: true,
    });

    const summary = await group.experiment("summary-approach", {
      variants: {
        map_reduce: async () => {
          const chunks = await step.run("chunk-document", () =>
            chunkText(extraction.result)
          );
          const partials = await step.run("summarize-chunks", () =>
            Promise.all(chunks.map((c) => summarize(c)))
          );
          return await step.run("combine-summaries", () =>
            combineSummaries(partials)
          );
        },
        single_pass: () =>
          step.run("single-pass-summary", () =>
            summarize(extraction.result)
          ),
      },
      select: experiment.bucket(`${userId}:summary`),
      withVariant: true,
    });

    return { extraction, summary };
  }
);
```

Appending a feature-specific suffix to the bucket key (`userId:extraction` vs `userId:summary`) lets the same user land on different variants in each experiment independently.

## Plug in your own selection logic

When the choice depends on something Inngest doesn't know about — a feature flag, an entitlement, an external service — use `experiment.custom()`. The function runs inside a memoized step, so the result is still deterministic across retries. It must return a string matching one of your variant names.

```ts
const recommendations = await group.experiment("recommendation-model", {
  variants: {
    collaborative: () =>
      step.run("collaborative", () => collaborativeFilter(userId)),
    mlBased: () =>
      step.run("ml-based", () => mlRecommend(userId)),
  },
  select: experiment.custom(async () => {
    const flag = await launchDarkly.variation("recommendation-model", { key: userId });
    return flag === "ml" ? "mlBased" : "collaborative";
  }),
});
```

## Gradually roll out a change

Experiments aren't only for model comparisons. Use `weighted` to roll out a rewritten implementation from 1% to 100%, watching for regressions in the dashboard at each stage. To change the rollout, deploy new weights: in-progress runs keep the variant they already selected, and only new runs use the updated split.

```ts
const invoice = await group.experiment("invoice-generation", {
  variants: {
    v1: () => step.run("generate", () => generateInvoiceV1(event.data)),
    v2: async () => {
      const lineItems = await step.run("fetch-items", () =>
        fetchLineItems(event.data.accountId)
      );
      const tax = await step.run("calculate-tax", () =>
        calculateTax(lineItems, event.data.region)
      );
      return await step.run("render", () => renderInvoiceV2(lineItems, tax));
    },
  },
  // Start at 1%, raise as confidence grows.
  select: experiment.weighted({ v1: 99, v2: 1 }),
});
```

Weights are relative and work for any number of variants. A `0` weight is valid and excludes a variant from new runs, which is how you finish a rollout (`{ v1: 0, v2: 100 }`). Weights that are all zero, negative, or non-finite throw when you create the strategy.

<Callout>
  With `experiment.weighted()`, selection happens over the weights map you pass — not the full variant list. Every variant you want eligible **must appear in the weights map**; a variant defined in `variants` but missing from `weights` is silently never selected. (`experiment.bucket()` differs: omit its weights and it splits evenly across all variant names.)
</Callout>

<Callout>
  Every variant callback must invoke at least one `step.run()` (or other `step.*`) call. The SDK throws a `NonRetriableError` if a variant completes without calling any step tools — code without steps isn't durable and would silently re-execute on every replay.
</Callout>

## Limitations

- **One variant per run.** Each run executes exactly one variant. Running all variants simultaneously (dual-running) is planned but not available yet. If you need to run every variant on the same input today — for example to diff their outputs directly — call the steps yourself outside `group.experiment()`. You lose the per-variant dashboard aggregation, but you get every result in one run.
- **Defined in code.** There's no UI for creating or editing experiments — you change configuration by deploying new code.
- **No automatic promotion.** You decide the winner and update the strategy yourself; Inngest won't promote a variant for you.
- **TypeScript only.** Python and Go SDK support is coming. If you're on the Python or Go SDK today, track support on the [roadmap](https://roadmap.inngest.com/roadmap?ref=docs) — there's no `group.experiment()` equivalent to fall back on yet.

For the full API surface, parameters, and selection-strategy details, see the [`group.experiment()` reference](/docs/reference/typescript/v4/functions/group-experiment).
