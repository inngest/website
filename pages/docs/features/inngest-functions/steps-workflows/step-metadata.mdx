# Step Metadata <VersionBadge version="TypeScript only" />

Step Metadata lets you attach custom key-value data to function runs, steps, and step attempts. This data is visible in the Inngest dashboard traces view, making it useful for tracking processing status, recording business-level metrics, or annotating runs with contextual information.

## Setup

Step Metadata requires the `metadataMiddleware()` to be added to your Inngest client:

```ts
import { Inngest } from "inngest";
import { metadataMiddleware } from "inngest";

const inngest = new Inngest({
  id: "my-app",
  middleware: [metadataMiddleware()],
});
```

Once enabled, `step.metadata()` becomes available inside your function handlers and `inngest.metadata` becomes available on the client instance.

## Basic usage

### Inside a step (recommended)

The primary way to attach metadata is via `step.metadata(memoId)` inside your function body. The `memoId` is a unique step ID used for memoization, ensuring the metadata update is only performed once even if the function retries.

When called inside `step.run()`, metadata is batched with the step's output -- making it **durable**. The metadata persists even if the function retries.

```ts
const processOrder = inngest.createFunction(
  { id: "process-order", triggers: [{ event: "shop/order.created" }] },
  async ({ event, step }) => {
    const charge = await step.run("process-payment", async () => {
      return await chargeCustomer(event.data.customerId, event.data.amount);
    });

    // Attach metadata to the current step attempt (durable)
    await step.metadata("record-payment-info").update({
      paymentId: charge.id,
      processor: "stripe",
      amount: event.data.amount,
    });

    await step.run("send-confirmation", async () => {
      await sendEmail(event.data.email, charge.receiptUrl);
    });
  },
);
```

### Client-level access

You can also use `inngest.metadata` to update metadata from outside a function body -- for example, from a separate API route that knows the run ID:

```ts
// Update metadata for a specific run from an external context
await inngest.metadata.run(runId).update({
  approvedBy: "admin@example.com",
  approvalTimestamp: Date.now(),
});
```

<Warning>
  Calling `metadata.update()` inside a function body but outside of a `step.run()` callback sends the update via a REST API call. This metadata is **not durable** and may be lost if the function retries. For durable metadata, always wrap updates in a step using `step.metadata()`.
</Warning>

## Scoping

Metadata can be attached at different levels of granularity using the builder pattern. Each method narrows the scope:

| Scope | Builder method | Description | Default when... |
|-------|---------------|-------------|-----------------|
| `run` | `.run(id?)` | Attached to the entire function run | Outside `step.run()` |
| `step` | `.step(id?, index?)` | Attached to a specific step | -- |
| `step_attempt` | `.attempt(index?)` | Attached to a specific step attempt | Inside `step.run()` |
| `extended_trace` | `.span(id)` | Attached to an extended trace span | -- |

### Auto-detection

When you call `step.metadata(id).update(values)` without explicitly setting a scope, the builder automatically detects your context:

- **Inside `step.run()`**: defaults to `step_attempt` scope for the current step and attempt
- **Outside `step.run()`** (function body): defaults to `run` scope

### Explicit scoping examples

```ts
const myFunction = inngest.createFunction(
  { id: "my-function", triggers: [{ event: "app/task.started" }] },
  async ({ event, step }) => {
    // Scope to the current run
    await step.metadata("run-status").run().update({
      priority: "high",
    });

    // Scope to a specific step
    await step.metadata("step-annotation").step("process-data").update({
      recordsProcessed: 1500,
    });

    // Scope to a specific step attempt (attempt index 0)
    await step.metadata("attempt-info")
      .step("process-data")
      .attempt(0)
      .update({
        startedAt: Date.now(),
      });

    // Scope to an extended trace span
    await step.metadata("span-details").span(spanId).update({
      externalCallDuration: 230,
    });
  },
);
```

The builder methods are chainable and each one removes itself from the available methods, preventing invalid combinations (e.g., you can't call `.run()` after `.step()`).

## Custom kinds

The second argument to `.update()` specifies a metadata **kind**. Kinds are used to group related metadata together. When multiple updates share the same kind and scope, their values are merged.

```ts
// Default kind: stored as "userland.default", displayed as "User Metadata"
await step.metadata("id-1").update({ status: "processing" });

// Custom kind: stored as "userland.billing", displayed as "User Metadata (billing)"
await step.metadata("id-2").update(
  { invoiceId: "inv_123", total: 99.99 },
  "billing",
);

// Another custom kind: stored as "userland.analytics", displayed as "User Metadata (analytics)"
await step.metadata("id-3").update(
  { source: "organic", campaign: "spring-2025" },
  "analytics",
);
```

### Built-in metadata kinds

Inngest automatically attaches system metadata to steps when applicable. These are read-only and managed by the platform:

| Kind | Label in dashboard | Description |
|------|-------------------|-------------|
| `inngest.ai` | AI Metadata | LLM/AI gateway data: input/output tokens, model, latency, cost |
| `inngest.http` | HTTP Metadata | HTTP request details: method, status code, request/response sizes |
| `inngest.http.timing` | HTTP Timing | Timing breakdown: DNS, TCP, TLS, TTFB, transfer |
| `inngest.response_headers` | Response Headers | HTTP response header key-value pairs |
| `inngest.warnings` | Warnings | Warning messages from the execution engine |

## Batching multiple updates

Use `.do()` to send multiple metadata updates within a single memoized step. This is useful when you need to set several kinds or scopes at once:

```ts
await step.metadata("update-all-statuses").do(async (builder) => {
  await builder.update({ phase: "started", initiator: "system" }, "workflow");
  await builder.update({ estimatedDuration: 30 }, "timing");
  await builder.update({ region: "us-east-1" }, "infra");
});
```

All updates inside `.do()` share the same memoization step, so they are all performed exactly once.

## Cross-run metadata

You can update metadata on a different run -- for example, a child function notifying its parent run of completion:

```ts
const childFunction = inngest.createFunction(
  { id: "child-process", triggers: [{ event: "workflow/child.started" }] },
  async ({ event, step }) => {
    const result = await step.run("do-work", async () => {
      return await processData(event.data);
    });

    // Update the parent run's metadata from the child function
    await step.metadata("notify-parent")
      .run(event.data.parentRunId)
      .update({
        childCompleted: true,
        resultCount: result.count,
        completedAt: Date.now(),
      });
  },
);
```

When targeting a different run, the update is always sent via the REST API (not batched), since the target run has a different execution context.

## Size limits

Metadata is subject to the following size limits, enforced server-side:

| Limit | Maximum size | Description |
|-------|-------------|-------------|
| **Per update** | **64 KB** | Maximum size of a single metadata update (span) |
| **Per run** | **1 MB** | Cumulative size of all metadata across the entire function run |

### How size is calculated

Size is calculated as the sum of each key's byte length plus the JSON-serialized byte length of each value:

```
size = sum(len(key) + len(JSON.stringify(value))) for each entry
```

For example, `{ "status": "ok" }` costs approximately `6 + 4 = 10 bytes` (key `"status"` = 6 bytes, value `"ok"` serialized = 4 bytes including quotes).

### Error handling

When a size limit is exceeded, the server returns an HTTP `413` error:

| Error | Message |
|-------|---------|
| Per-update exceeded | `"Metadata span exceeds maximum size of 64KB"` |
| Per-run exceeded | `"Cumulative metadata size exceeds limit"` |

<Info>
  If you're working with large metadata payloads, consider storing the data externally (e.g., in a database or object store) and attaching only a reference ID or URL as metadata.
</Info>

Step metadata is displayed in the **trace detail panel** when you select a span in the function run timeline.

## How it works

Understanding the data flow helps when debugging or optimizing metadata usage.

### Inside `step.run()` (batched path)

1. Your code calls `step.metadata(id).update(values)`
2. The metadata is collected in-memory alongside the step's output
3. When the step completes, both the step result and metadata are sent to the Inngest server in a single response
4. The server validates the metadata, enforces size limits, and creates OpenTelemetry spans
5. Metadata spans are stored in the trace backend and become visible in the dashboard

### Outside `step.run()` (API path)

1. Your code calls `step.metadata(id).update(values)` or `inngest.metadata.update(values)`
2. A REST API call is made to `POST /v1/runs/{runID}/metadata`
3. The server loads the run's existing metadata to check cumulative size
4. After validation, the server creates metadata spans and persists the size delta

```
SDK (your code)
  |
  |-- Inside step.run() --> Batched with step output --> Server
  |
  |-- Outside step.run() --> REST API call ------------> Server
                                                           |
                                                     Validate & enforce limits
                                                           |
                                                     Create OTel spans
                                                           |
                                                     Store in trace backend
                                                           |
                                                     Display in dashboard
```

## Limitations

- **TypeScript SDK only** -- Python and Go SDK support is planned but not yet available.
- **Durability requires `step.run()`** -- Metadata updates outside of a step context are sent via REST and may be lost on function retries.
- **Merge-only** -- The only supported operation is `merge`, which combines values of the same kind. You cannot delete or replace individual metadata keys.
- **Unique memoization IDs** -- Each `step.metadata(memoId)` call creates an internal step. The `memoId` must be unique within the function, just like any other step ID.
- **Server-side enforcement** -- Size limits are enforced by the server, not the SDK. Exceeding limits returns HTTP 413 errors that surface as exceptions in your function.

## API reference

### `step.metadata(memoId)`

Creates a durable metadata update wrapped in a step. Requires `metadataMiddleware()` in your client middleware.

| Method | Signature | Description |
|--------|-----------|-------------|
| `.run(id?)` | `(id?: string) => MetadataBuilder` | Scope to a run. Omit `id` for the current run. |
| `.step(id?, index?)` | `(id?: string, index?: number) => MetadataBuilder` | Scope to a step. Omit `id` for the current step. |
| `.attempt(index?)` | `(index?: number) => MetadataBuilder` | Scope to a step attempt. Omit `index` for the current attempt. |
| `.span(id)` | `(id: string) => MetadataBuilder` | Scope to an extended trace span. |
| `.update(values, kind?)` | `(values: Record<string, unknown>, kind?: string) => Promise<void>` | Send the metadata update. `kind` defaults to `"default"`. |
| `.do(fn)` | `(fn: (builder: MetadataBuilder) => Promise<void>) => Promise<void>` | Batch multiple updates in a single memoized step. |

### `inngest.metadata`

Client-level metadata builder. Same methods as above except `.do()`. Requires `metadataMiddleware()`.

| Method | Signature | Description |
|--------|-----------|-------------|
| `.run(id?)` | `(id?: string) => MetadataBuilder` | Scope to a run. |
| `.step(id?, index?)` | `(id?: string, index?: number) => MetadataBuilder` | Scope to a step. |
| `.attempt(index?)` | `(index?: number) => MetadataBuilder` | Scope to a step attempt. |
| `.span(id)` | `(id: string) => MetadataBuilder` | Scope to an extended trace span. |
| `.update(values, kind?)` | `(values: Record<string, unknown>, kind?: string) => Promise<void>` | Send the metadata update. |
