Use deferred scoring to grade an agent on what actually happened: human feedback, a resolved ticket, or ground truth that only arrives after the run ends.

The most honest signal for an agent is whether the real-world outcome was good, and that usually isn't known when the run finishes. 

A triage agent posts a root-cause analysis now; whether it pointed at the right file is only known once an engineer ships the fix. 

A support agent answers now; whether it resolved the ticket is known hours later.

Because Inngest runs the agent, you can attach a score to that original run whenever the truth arrives; that can be inline if it's known during the run, or deferred against the run's ID if it lands later. The score lives next to the run, step, and trace that produced it, with no separate eval pipeline to ship traces to.

This pattern is useful when you want to grade against a resolved outcome, capture a human 👍/👎, compare an agent's claim to ground truth, or measure goal completion rather than a model-judged proxy.

## Which kind of score do you need?

| Situation | Approach | Why |
| --- | --- | --- |
| The outcome is known during the run | `step.score()` inline | Score the run as it completes. |
| The outcome arrives later (feedback, resolution, ground truth) | `inngest.score()` against the `runId` | Attach the score to the original run when the truth lands. |
| You want the same scoring logic in several places | `createScorer()` | Define a named scorer once, reuse it. |
| You're comparing variants | Pair with `group.experiment()` | The same scorer grades every variant. |

A score is a number (commonly `0`–`1`) attached to a run, a specific step, or a session. You decide what it measures.

## Score inline when the outcome is known

If the agent can be graded against a known answer at the time it runs, whether a fixed value range or a deterministic algorithm, score it inline with `step.score()`. The score is memoized like any other step.

```typescript
export default inngest.createFunction(
  { id: "classify-ticket", triggers: { event: "support/ticket.created" } },
  async ({ event, step }) => {
    const predicted = await step.run("classify", () =>
      classify(event.data.body)
    );

    if (event.data.knownLabel) {
      await step.score({
        name: "classification-accuracy",
        value: predicted === event.data.knownLabel ? 1 : 0,
      });
    }

    return predicted;
  }
);
```

## Defer the score until the real outcome lands

This is the common case for agents in production: the run finishes, and the outcome is only known later. Let the run complete, then score it from a second function triggered by the outcome event, referencing the original `runId`.

In the incident-triage example, the agent posts a root-cause analysis and cites the files it believes are responsible. When the fix actually ships, you compare the cited files to the files the fix touched, and attach that score to the original run.

```typescript
// 1. The agent run records the run ID where the outcome can find it later.
export default inngest.createFunction(
  { id: "triage-incident", triggers: { event: "incident/opened" } },
  async ({ event, step, runId }) => {
    const rca = await step.run("investigate", () => investigate(event.data));
    await step.run("post-rca", () => postRca(event.data.incidentId, rca));
    await step.run("link-run", () =>
      incidents.attachRunId(event.data.incidentId, runId)
    );
    return rca;
  }
);

// 2. When the fix ships, grade the original run against ground truth.
export const scoreTriage = inngest.createFunction(
  { id: "score-triage", triggers: { event: "incident/fix.shipped" } },
  async ({ event, step }) => {
    const { runId, citedFiles } = await step.run("load", () =>
      incidents.load(event.data.incidentId)
    );

    await inngest.score({
      name: "localization",
      runId,
      value: jaccard(citedFiles, event.data.fixFiles), // overlap, clamped 0–1
    });
  }
);
```

The score now sits on the triage run, so the run's trace, cost, and quality are all in one place. If no outcome event is coming and you need to grade after a delay, schedule the scoring work with `defer()` instead of waiting on an event.

## Capture human feedback as a score

A thumbs-up/down on an agent's output is an outcome signal. Send it as an event from your UI and score the run it refers to.

```typescript
export const scoreFeedback = inngest.createFunction(
  { id: "score-feedback", triggers: { event: "agent/response.rated" } },
  async ({ event }) => {
    await inngest.score({
      name: "human-feedback",
      runId: event.data.runId,
      value: event.data.rating === "up" ? 1 : 0,
    });
  }
);
```

## Score the session, not just the run

A conversation or an incident thread spans many runs. When the outcome reflects the whole flow rather than one step, score the session instead of a single run, using the same session key you group runs by.


```typescript
await inngest.score("goal-completed", {
  session: { conversation_id: event.data.conversationId },
  value: event.data.resolved ? 1 : 0,
});
```

## Reuse scoring logic with a scorer

When the same grading logic shows up in more than one place, define it once with `createScorer()` (imported from `inngest/experimental`) and reuse it. A scorer runs as its own function: it takes input matching its schema and returns a `{ name, value }` score.

```typescript
import { createScorer } from "inngest/experimental";
import { z } from "zod";

export const localizationScorer = createScorer(
  inngest,
  {
    id: "localization-scorer",
    schema: z.object({ cited: z.array(z.string()), fix: z.array(z.string()) }),
  },
  async ({ event }) => {
    const { cited, fix } = event.data.input;
    return { name: "localization", value: jaccard(cited, fix) };
  }
);
```

Trigger it from any function with `defer()`, passing data that matches the scorer's schema:

```typescript
defer("score-localization", {
  function: localizationScorer,
  data: { cited: citedFiles, fix: event.data.fixFiles },
});
```

## Best practices

- Score the outcome you care about, not a proxy for it. Prefer "did the ticket resolve" over "did the answer look good."
- Use a stable score name. Names appear in the dashboard and in Insights, and renaming splits your history.
- Keep `value` in `[0, 1]`. Map pass/fail to `1`/`0` and graded results to a fraction.
- Record the `runId` somewhere durable if the outcome arrives later, so the deferred scorer can find the run it belongs to.
- Reach for a deferred score before an LLM-as-judge score. A real outcome is a better signal than a model's opinion of the output.
