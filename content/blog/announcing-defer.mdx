Orchestration tools are great at running one thing after another — less so when the follow-up work needs to know exactly what happened *during* the function, not just what triggered it.

Take a scorer for an AI response. It needs the prompt, the retrieved documents, the model version, and the latency — none of which is in the trigger event. Today, getting that context into a follow-up handler means either padding the trigger event with everything the handler might need, or writing the parent's intermediate state to your database and reading it back. Either way, you're maintaining a contract between two functions that don't know about each other, and keeping it in sync as the parent changes. And that's before you start scoring against signals that haven't arrived yet: a user conversion days out, a thumbs up that may never come.

`defer()` is a new Inngest API for exactly this. You define follow-up work as a typed, durable Inngest function and launch it directly from the parent — passing exactly the data it needs. Reach for it when work should happen *because of* a run, not *as part of* it: scoring an agent's output, sending a notification, logging a side effect, or queueing follow-up work. Each deferred run is its own durable function with separate retries and full access to `step.run`, `step.waitForEvent`, and `step.sleep`. It's scoped to the parent run in the UI so you can see what fired what.

This capability is only possible because all Inngest functions run directly from your codebase. The deferred function is right next to the one that launches it, the payload contract is enforced at the type level, and refactors keep working because you're passing a function reference — not matching on an event name string.

Learn exactly how it works, and when to use it, below.

> `defer()` is available today as an experimental API. `createDefer` is imported from `inngest/experimental` and the surface may change before GA.

## Why is this a problem right now?

Post-completion background work is not a new problem, but it's becoming a more painful one. Business logic explodes in multi-step durable workflows, and the surface area of "things that should happen because of a run" keeps widening. There are only a few ways to solve this today, and they all share the same drawback: the follow-up is disconnected from the work that triggered it.

**Traditional job queues** let you enqueue work after a function, but the queued job is a separate handler in a separate file. You define a payload contract by convention, hope the producer and consumer agree on its shape, and lose any visual link between the original run and its follow-up.

**Workflow engines** give you failure callbacks and child workflows, but the wiring still lives outside your function — another file, another event shape, another place a refactor can quietly drift.

**Eval platforms** put the scorer in a separate system from the function it's scoring. The feedback loop between "I shipped a change" and "I understand how it performed" runs through a second SDK, a second deployment, and a second source of truth.

What none of these approaches offer is a typed, single-target, durable follow-up that you can launch in one line from the function that produced the data — and see linked back to that run in the UI.

## How `defer()` works

You define the deferred function once with `createDefer`, register it alongside your other functions in `serve()`, and call it from any function via the `defer` context method:

```tsx
import { createDefer } from "inngest/experimental";
import { z } from "zod";

export const trackAnalytics = createDefer(
  inngest,
  {
    id: "track-analytics",
    schema: z.object({ orderId: z.string(), userId: z.string() }),
  },
  async ({ event, step }) => {
    await step.run("send-to-segment", () =>
      analytics.track("order_created", {
        orderId: event.data.orderId,
        userId: event.data.userId,
      })
    );
  }
);

export const processOrder = inngest.createFunction(
  { id: "process-order", triggers: { event: "order/created" } },
  async ({ defer, event, step }) => {
    const orderId = await step.run("create-order", () =>
      db.orders.create(event.data)
    );

    const charge = await step.run("charge-payment", () =>
      stripe.charges.create({ amount: event.data.total, orderId })
    );

    defer("track-order-analytics", {
      function: trackAnalytics,
      data: { orderId, userId: charge.user.id },
    });

    return orderId;
  }
);
```

When the parent run finalizes, the deferred callback fires as its own linked Inngest run — durable, observable, retryable — with the exact payload you passed in.

**Deferred runs are separate, linked runs.** Each `defer()` call produces its own run, connected to the parent. It has its own `runId`, its own step history, its own retry behavior. They don't pollute your runs list; deferred runs are scoped to the parent run's detail view.

**`defer()` is fire-and-forget.** The call returns `void` and the parent run continues immediately. Buffered defer ops ship when the parent run finalizes, so the deferred function can't block, fail, or change the parent's outcome.

**Steps work inside deferred runs.** The full power of Inngest — `step.run()`, `step.waitForEvent()`, `step.sleep()` — is available inside the deferred handler. This is what makes patterns like outcome-based scoring possible.

**Payloads are typed.** When you pass a `schema` to `createDefer`, the `data` you send is validated at the call site and on the receiver, and `event.data` is typed in the handler. Refactoring the payload becomes a type error at every call site rather than a runtime surprise.

**IDs must be unique within the parent run.** Unlike step IDs, no implicit index is appended. This is intentional: `defer()` works inside `step.run()`, and an implicit index would change on re-entry. If you call `defer()` with the same ID twice, the duplicate is logged and dropped.

You already have ways to launch work from inside a function. `step.invoke` runs another function and waits for the result — useful when you need the return value, but the parent blocks until it finishes. `step.sendEvent` is fire-and-forget, but broadcasts to any function whose trigger matches. And a plain `step.run` becomes *part of* the parent run, with its retries bundled in and visible in the parent's step list. `defer` fills the gap. The call itself is fire-and-forget like `sendEvent`, but the target is one specific typed function like `invoke`. And unlike any of them, the run it produces is linked back to the parent in the UI, so you can find it later without instrumenting anything yourself.

## When to use defer()

### Scoring AI responses after serving them

You ran an LLM call and served a result to the user. Now you want to evaluate quality programmatically without blocking the response — and without re-fetching everything that went into producing the output. With `defer`, the scorer is a function in your codebase, deploying alongside everything else and observed by the same tooling. No second SDK, no separate deployment surface, no extra identifier to thread through your database so a later signal can be joined back.

```tsx
export const scoreResponse = createDefer(
  inngest,
  {
    id: "score-response",
    schema: z.object({
      runId: z.string(),
      prompt: z.array(z.object({ role: z.string(), content: z.string() })),
      response: z.string(),
    }),
  },
  async ({ event, step }) => {
    const score = await step.run("run-eval", () =>
      evaluator.score({
        prompt: event.data.prompt,
        response: event.data.response,
        criteria: ["relevance", "accuracy", "tone"],
      })
    );

    await step.run("persist-score", () =>
      db.scores.create({ runId: event.data.runId, score })
    );
  }
);

export const generateReply = inngest.createFunction(
  { id: "generate-reply", triggers: { event: "chat/message.received" } },
  async ({ defer, event, runId, step }) => {
    const completion = await step.run("generate-response", () =>
      openai.chat.completions.create({
        messages: event.data.messages,
        model: "gpt-4o",
      })
    );

    const response = completion.choices[0].message.content;

    defer("score", {
      function: scoreResponse,
      data: { runId, prompt: event.data.messages, response },
    });

    return response;
  }
);
```

Because the deferred run is a full durable function, the scorer can also `step.waitForEvent()` — waiting days for a conversion signal, a thumbs up, or any downstream event, and then scoring retroactively with that richer signal.

### Outcome-based scoring (or "wait for conversion")

Some signals don't show up for days. A user converts, a support ticket gets reopened, a recommendation pays off. You can't block a parent run for a week waiting on that. With `defer`, the deferred run itself does the waiting — pausing on `step.waitForEvent` for as long as you need before recording the outcome.

```tsx
export const conversionScore = createDefer(
  inngest,
  {
    id: "conversion-score",
    schema: z.object({
      runId: z.string(),
      userId: z.string(),
      recommendedProductId: z.string(),
      recommendedAt: z.string(),
    }),
  },
  async ({ event, step }) => {
    const conversion = await step.waitForEvent("wait-for-purchase", {
      event: "purchase/completed",
      timeout: "7d",
      if: `async.data.userId == '${event.data.userId}'`,
    });

    const converted = conversion !== null;

    await step.run("record-outcome", () =>
      db.scores.create({
        runId: event.data.runId,
        recommendation: event.data.recommendedProductId,
        converted,
        daysToConvert: converted
          ? daysBetween(event.data.recommendedAt, conversion.ts)
          : null,
      })
    );
  }
);
```

The scorer pauses for up to seven days for a matching `purchase/completed` event, records the outcome whether or not it arrives, and stays linked to the run that made the original recommendation.

### Cache invalidation after a write

Register invalidation inline with the write, so it's hard to forget and easy to find when you're reading the parent:

```tsx
export const invalidateProductCache = createDefer(
  inngest,
  {
    id: "invalidate-product-cache",
    schema: z.object({ productId: z.string() }),
  },
  async ({ event, step }) => {
    await step.run("clear-cdn", () =>
      cdn.purge(`/products/${event.data.productId}`)
    );
    await step.run("clear-search-index", () =>
      searchIndex.reindex(event.data.productId)
    );
  }
);

export const updateProduct = inngest.createFunction(
  { id: "update-product", triggers: { event: "product/updated" } },
  async ({ defer, event, step }) => {
    await step.run("update-product", () =>
      db.products.update(event.data.productId, event.data.changes)
    );

    defer("invalidate", {
      function: invalidateProductCache,
      data: { productId: event.data.productId },
    });
  }
);
```

### Sending analytics without blocking

You want rich analytics — including state that was only assembled mid-function — without holding up the response:

```tsx
export const trackRecommendationServed = createDefer(
  inngest,
  {
    id: "track-recommendation-served",
    schema: z.object({
      userId: z.string(),
      plan: z.string(),
      recommendationCount: z.number(),
      modelVersion: z.string(),
      latencyMs: z.number(),
    }),
  },
  async ({ event, step }) => {
    await step.run("send-analytics", () =>
      analytics.track("recommendations_served", event.data)
    );
  }
);

export const serveRecommendations = inngest.createFunction(
  { id: "serve-recommendations", triggers: { event: "recs/requested" } },
  async ({ defer, event, step }) => {
    const user = await step.run("fetch-user", () =>
      db.users.findById(event.data.userId)
    );

    const recommendations = await step.run("generate-recs", () =>
      recommender.generate(user)
    );

    defer("track", {
      function: trackRecommendationServed,
      data: {
        userId: user.id,
        plan: user.plan,
        recommendationCount: recommendations.items.length,
        modelVersion: recommendations.modelVersion,
        latencyMs: recommendations.latencyMs,
      },
    });

    return recommendations;
  }
);
```

The parent returns immediately. The analytics call runs in the background with the full context it needs, as a durable run you can inspect from the parent's detail view.

## A new primitive, not just a feature

We built `defer()` as a primitive. The underlying mechanism is a full durable run with access to the rest of the Inngest API surface, which means there's more coming. The feedback loop between shipping logic and understanding how something actually performed should be as tight as possible.

## Getting started

`defer()` is available today as an experimental API in the Inngest TypeScript SDK. Check the [deferred functions documentation](/docs/features/inngest-functions/steps-workflows/deferred-functions) for the full API reference.

We're especially interested in hearing how teams use this for durable workflows — including ensuring agents run as expected. If you're building eval pipelines or scoring loops and want to talk through how `defer()` fits, reach out.
