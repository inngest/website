Python's `asyncio` gives you `Event`, `Condition`, and `Queue` for coordinating between tasks. None of them do what we actually needed: let multiple consumers wait for a value to reach a specific state.

We wrote `ValueWatcher` to fill that gap. It's generic, fully type-safe, and about 100 lines of code. It replaced a lot of fragile callback wiring in our WebSocket connection manager.

</Callout>

The property we care about most: no lost updates. In the example below, two consumers wait for different values, and both values are set synchronously without yielding to the event loop. Both consumers complete.

```python
watcher = ValueWatcher(0)

# One consumer waits for 1, another waits for 2.
wait = await start_tasks(
    watcher.wait_for(1),
    watcher.wait_for(2),
)

# Set both values without yielding.
watcher.value = 1
watcher.value = 2

# Both consumers complete.
await wait()
```

This is the property that makes `ValueWatcher` trustworthy in a system where state changes can happen faster than consumers can process them.

## The problem

Our Connect protocol manages a WebSocket connection with a lot of concurrent async handlers: heartbeats, execution dispatch, reconnection logic, handshakes, etc. These handlers need to react to shared state changes. For example:

- Wait for the WebSocket connection to be established before sending messages
- Wait for the handshake to complete before dispatching work
- Wait for all pending requests to drain before closing

The naive approach is `asyncio.Event`, but events are boolean. If you need to wait for a value to become `3` or stop being `None`, you need multiple events and careful coordination between them. That coordination is where bugs live.

## The solution

`ValueWatcher` wraps a value and lets consumers block until it meets some condition:

```python
state = ValueWatcher(ConnectionState.CONNECTING)

# In the reconnect handler:
await state.wait_for(ConnectionState.CONNECTED)

# In the close handler:
await state.wait_for_not(ConnectionState.CONNECTING)

# In a handler that needs the WebSocket:
ws_watcher = ValueWatcher[websockets.ClientConnection | None](None)
ws = await ws_watcher.wait_for_not_none()
```

Setting the value is just assignment:

```python
state.value = ConnectionState.CONNECTED
```

All consumers are notified immediately.

## How it works

Each call to `wait_for` (or its variants) creates a dedicated `asyncio.Queue`. When the value is set, the setter iterates over all queues and pushes the `(old, new)` tuple into each one. The waiting coroutine pulls from its queue and checks the condition.

```python
@value.setter
def value(self, new_value: T) -> None:
    if new_value == self._value:
        return

    old_value = self._value
    self._value = new_value

    for queue in self._watch_queues:
        queue.put_nowait((old_value, new_value))
```

Per-consumer queues are the key design choice. A single `asyncio.Event` would lose intermediate transitions — if the value goes from `0` to `1` to `2` before a consumer runs, it only sees `2`. With queues, every transition is buffered and delivered in order. This matters when you have one consumer waiting for `1` and another waiting for `2`.

The queue cleanup uses a context manager:

```python
async def wait_for(
    self,
    value: T,
    *,
    immediate: bool = True,
    timeout: float | None = None,
) -> T:
    if immediate and self._value == value:
        return self._value

    async def _wait() -> T:
        with self._watch() as watch:
            # Re-check after queue registration.
            if immediate and self._value == value:
                return self._value

            async for _, new in watch:
                if new == value:
                    return new

    return await asyncio.wait_for(_wait(), timeout=timeout)
```

<Callout>
  The `immediate` parameter controls whether `wait_for` returns instantly if the value already matches. This defaults to `True`, which is usually what you want. Setting it to `False` forces the caller to wait for the _next_ transition to that value, even if it's the current value. This is useful when you need to observe an actual change, not just the current state.
</Callout>

`_watch()` returns a context manager that registers a queue on enter and removes it on exit. The context manager yields an async generator that pulls from the queue. When the `wait_for` coroutine returns (or the task is cancelled), the `with` block exits and the queue is cleaned up.

## A typing trick

`wait_for_not_none` narrows the return type by constraining `self`:

```python
async def wait_for_not_none(
    self: ValueWatcher[S | None],
    *,
    immediate: bool = True,
) -> S:
```

This means `ValueWatcher[str | None].wait_for_not_none()` returns `str`, not `str | None`. The method is only callable on watchers whose type parameter is an optional. This is checked at type-check time — no runtime cost.

## Why not X?

### `asyncio.Event`

`asyncio.Event` is a boolean flag that can be set and cleared. We need to support arbitrary types, not just booleans.

### `asyncio.Condition`

`asyncio.Condition` can be used to wait for a value to meet a condition, but it has limitations.

Beyond the clunky API (manual lock management, re-checking after spurious wakeups), `Condition` has a fundamental problem with intermediate transitions. `Condition.wait_for(predicate)` re-evaluates the predicate against the _current_ value when the consumer wakes. If the value transitions from `0` to `1` to `2` before a consumer waiting for `1` runs, the predicate `value == 1` fails (it's `2` now) and the consumer blocks forever. `ValueWatcher` solves this with per-consumer queues that buffer every transition — no intermediate state is lost.

### Callbacks

Callbacks push complexity to the consumer. Instead of linear control flow logic, you have to manage callbacks and their state. `ValueWatcher` keeps the control flow linear and predictable.

### Third-party reactive libraries

Too heavy for what we need. We don't want a dependency for a 100-line utility.

## Limitations

The setter deduplicates by equality: if the new value `==` the current value, no notification fires. This is convenient for enums and primitives, but it uses `==`, not `is`. If you store a mutable object, mutate it in place, and reassign the same reference, the check passes and consumers won't be notified. Stick to immutable values (enums, strings, ints, frozen dataclasses) and this isn't a concern.
